# Contributing

This is a solo research project and I welcome scrutiny more than I welcome help — though both are good.

## The most valuable contributions right now

1. **Finding errors.** If a test doesn't verify what it claims, if a number in the README doesn't match the code, if a mathematical claim is wrong — open an issue. This is the highest-value thing anyone can do.

2. **Running the code and reporting what happens.** Clone it, run the quick-start commands, tell me if something breaks on your machine. Especially valuable for Ghost Meadow hardware tests (see [HARDWARE_WANTED.md](HARDWARE_WANTED.md) if applicable).

3. **Asking hard questions.** "Why did you choose X over Y?" and "Doesn't Z already solve this?" are the kind of questions that improve the work. Use Discussions or Issues.

## If you want to submit code

- Open an issue first describing what you want to change and why
- Keep PRs small and focused on one thing
- Include a test or verification step if you're touching anything mathematical
- I'll review personally — this isn't a team with a review queue, so expect human latency

## Style

- Commit messages: plain English, 1-2 sentences, describe *what changed and why*
- Code: match the existing style in the file you're editing
- READMEs: state claims precisely, include limitations, no hype
